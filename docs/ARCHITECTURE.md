## Executive Summary

A single Next.js (App Router, TypeScript) application deployed on Vercel. The fan surface (/) is a mobile-first multilingual copilot: a streaming Gemini chat grounded on a static JSON venue knowledge base, with function calling into deterministic in-repo data services (directions over a routing graph, crowd levels from a seeded time-based simulation, transport options). Routes render as highlighted paths on a hand-authored SVG stadium map. All Gemini calls go through one server-side gateway that degrades to canned, non-AI responses on any failure, so the deployed demo never hard-fails. The ops surface (/ops) is a gated stretch that reuses the same simulation and Gemini gateway with JSON-mode outputs.

## Objectives

* Primary Objective: A non-local-speaking fan completes real in-stadium journeys (seat, food, exits, transport) in their own language.
* Secondary Objective: Demonstrate GenAI craft — grounding, function calling, JSON mode, fallbacks — over model size.
* Success Criteria: All FEATURES.md Musts pass their phase gates; deployed link works with zero env vars set.

## Data Flow

Request lifecycle for "¿Dónde está mi asiento? Sección 214":

1. Client POSTs `{messages, accessibilityMode, seat?}` to `/api/chat`; opens SSE stream.
2. Route handler assembles system prompt (venue KB summary + rules + language directive) and tool declarations from `lib/prompts.ts`.
3. Gemini streams; model emits `getDirections({to: "seat-214", accessible: false})` function call.
4. Handler executes `lib/venue.ts` route() → step list + polyline node ids; returns tool result to Gemini; Gemini continues streaming the Spanish answer.
5. Handler emits SSE events: `text` chunks + one `route` event; client renders steps in chat and highlights the path on `StadiumMap`.
6. On any Gemini error/timeout/missing key: gateway throws → handler answers from `lib/fallback.ts` (same SSE shape, `degraded: true` flag shown as a subtle badge).

## Database Overview

No database. Static JSON entities:

| Entity | Purpose |
| ------ | ------- |
| `venue.json` zones[] | id, name, level, polygon/svg ref, adjacent zone ids |
| `venue.json` pois[] | id, kind (gate/food/restroom/prayer/medical/exit/seating), name, tags (halal, accessible), nodeId |
| `venue.json` graph {nodes[], edges[]} | Walkable waypoints with x,y (SVG coords); edges with distance + `accessible` flag (stairs vs ramp/lift) |
| `matches.json` | Fixture, kickoff, phase timings (drives sim modifiers + transport suggestions) |
| `transport.json` | Metro/bus/parking options: frequency, walk time, post-match surge notes |
| `faqs.json` | Policy Q&A (bags, re-entry, water, tickets) for grounding |

## AI Workflow

Input (user message, any language)
↓
Prompt (system: role + venue KB digest + grounding rules + "reply in the user's language" + accessibility variant; few-shot for tool choice)
↓
Model (`gemini-2.5-flash`, streaming; tools: getDirections/getCrowdLevel/getTransport; ops routes use `responseMimeType: application/json` + responseSchema)
↓
Validation (tool args validated against typed schemas before execution; JSON-mode outputs parsed + schema-checked; on failure → one retry → fallback)
↓
Response (streamed text + structured events; degraded badge if fallback used)

Prompt principles (AI_ENGINEER): ground every factual claim in provided KB — instruct "if not in the data, say you don't know and suggest the info desk"; never invent gate/route facts (tool results are the only source of directions); keep system prompt < ~1.5k tokens by shipping a KB digest, with full detail behind tools; Arabic/Hindi tested explicitly (RTL rendering for AR).

## Error Handling Strategy

| Failure | Recovery |
| ------- | -------- |
| Missing `GEMINI_API_KEY` | Boot in degraded mode; banner "demo mode"; full non-AI navigation still works |
| Gemini 429/5xx | 1 retry with backoff (300ms) → fallback resolver |
| Gemini timeout (>8s first token) | Abort → fallback resolver |
| Bad tool args from model | Zod-style validation → return error message to model once → else fallback |
| Malformed JSON-mode output | Parse + schema check → 1 re-ask → canned ops sample |
| Unknown intent / out-of-venue question | Prompt instructs graceful redirect; fallback has generic multilingual "info desk" reply |
| Client SSE drop | UI shows retry button; messages kept client-side |

## Security Considerations

* Secrets: `GEMINI_API_KEY` server-side env only; never sent to client; `.env.example` committed
* Validation: all tool args and ops inputs validated server-side; message length caps
* Authentication: none (public demo, no user data stored)
* Rate Limits: per-IP soft cap in chat route (simple token bucket in module scope, best-effort) to protect quota during judging

## Deployment Strategy

- Development: `npm run dev`, `.env.local`
- Demo: Vercel — connect GitHub repo, set `GEMINI_API_KEY`, default build. Verify: deployed URL works with the env var removed (degraded mode check), then re-add
- Production (Optional): n/a for hackathon

Vercel notes: route handlers on Node runtime (streaming supported, no edge-runtime SDK surprises); `maxDuration = 30` for `/api/chat`; simulation is time-pure so serverless instances agree on crowd state.

## Trade-offs

| Decision | Benefit | Cost |
| -------- | ------- | ---- |
| Simulated venue data | Deterministic, demo-safe, zero external deps | Less "real-world" claim; documented honestly in README |
| Raw SSE over AI SDK | Full control of tool loop + custom map events | ~1h more plumbing |
| Single flash model | Speed, quota headroom, matches jury guidance | Slightly weaker reasoning on odd queries |
| Hand-authored SVG map | Precise routing visuals | Fictional stadium layout (~1.5h to author) |
| No DB | Zero infra risk | Ops incident feed is simulated, not persisted |
